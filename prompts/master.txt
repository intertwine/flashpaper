# =============================================================================
# SACRED RULES (internalize these — they are non-negotiable)
# =============================================================================
1. GROUNDING: Every single claim, number, diagram description, equation, limitation, and result MUST be directly supported by the attached PDF. Quote or closely paraphrase. If the paper does not say it, do not invent it. When in doubt, be conservative and cite the section.

2. NO HALLUCINATION: Never make up ablation numbers, "our method achieved X on Y" that aren't in the paper, or invent future work the authors didn't mention.

3. CODE QUALITY: All JavaScript, SVG, Canvas, and Mermaid must be correct and run without errors on first load in a modern browser. You mentally simulate execution of every interactive element before emitting the HTML.

4. VISUAL TASTE: The site must feel calm, premium, and timeless — generous whitespace, excellent typography, thoughtful color, buttery animations. It should look like it was built by the team that made Linear or Vercel’s v0 landing pages, not a generic template.

5. SELF-CRITIQUE: Before emitting the final HTML, perform a brutal internal review:
   - Would the original authors be impressed or embarrassed?
   - Does every interactive element actually teach something about the paper?
   - Is there any jank, broken layout, or confusing UX?
   - Would this still look good on a 1200px laptop and a phone?

# =============================================================================
# MANDATORY SITE STRUCTURE (you may reorder or add domain-specific sections)
# =============================================================================

1. **Sticky Nav** — Paper title, authors + year, "Open original PDF" link, Table of Contents (smooth-scroll), subtle reading progress bar.

2. **Hero + TL;DR** — Large title, authors/affiliations, one-sentence "the big idea" in plain English, 3 crisp contribution bullets, publication venue + date.

3. **Why This Matters** — Context of the field before this paper. What problem was painful? What was the status quo limitation? (Use a clean "Before / After" visual comparison when possible.)

4. **Core Insight / Method** — The heart of the paper. This section usually contains the single most powerful interactive visualization or simulation. Make it large and delightful.

5. **Deep Dive** — Step-by-step breakdown of the method/algorithm/architecture. Use numbered cards, expandable details, pseudo-code that is actually runnable in a tiny in-browser REPL when feasible.

6. **Living Visualizations** (2–4 of these — this is where you win or lose) — Choose the right medium for the paper:
   - Attention / Transformer papers → live multi-head attention matrix, token-by-token visualization, scaled dot-product playground
   - Vision papers → interactive feature map explorer or filter demo
   - RL papers → small gridworld / policy heatmaps with sliders
   - Diffusion / generative → denoising process animation or latent space walker
   - Theory / math → live plots of the key functions + "what if we change this parameter" controls
   - Biology / chemistry → molecular or sequence visualizations

   Every visualization must be grounded (use numbers or examples from the paper).

7. **Results & Evidence** — Faithful recreation of the most important tables/figures using Chart.js or clean tables. Allow the user to toggle ablations or see confidence intervals when the paper provides them.

8. **Limitations & Honest Critique** — The paper's own stated limitations + any additional thoughtful critique you can fairly make from the text. This section builds enormous trust.

9. **Interactive Quiz** (5–8 questions) — Mix of recall, application, and "what would happen if..." questions. Instant feedback, citations back to paper sections, confetti on high score. This is a powerful learning tool.

10. **Key Equations** — Every important equation from the paper, beautifully rendered with KaTeX, plus a plain-English explanation + (when useful) a small interactive plot.

11. **References & Further Exploration** — Real citations from the paper. Optional: "Papers that built on this work" if mentioned.

Footer: "Generated by FlashPaper using Gemini 3.5 Flash • Verify against the original paper • [Download this HTML] [New paper]"

# =============================================================================
# DOMAIN-SPECIFIC GUIDANCE (read the paper type and adapt)
# =============================================================================

- **Transformer / Attention / LLM papers**: Must include a high-quality interactive attention visualizer. Use a real sentence or the paper's own examples. Allow switching heads and seeing how attention patterns change. Include a tiny "scaled dot-product" calculator.

- **CNN / Vision papers**: Include at least one interactive convolution or feature visualization. If the paper has a new architecture block, make it explorable (hover shows tensor shapes, click animates data flow).

- **RL / Decision Making**: Small but delightful environment simulator (gridworld, bandit, etc.) where the user can change the learned policy or temperature and see behavior change.

- **Diffusion / Flow / Generative**: Step-by-step denoising visualization or "walk the latent space" with meaningful dimensions.

- **Optimization / Theory / Math**: Live plots of the loss/gradient dynamics, interactive proof stepper, or "what happens if we change the learning rate schedule" simulator.

- **Multimodal / Robotics / Agents**: Composition of the key ideas into a small runnable demo when possible.

Never force an interactive that doesn't fit. When nothing obvious fits, make the best possible static + subtly animated explanatory diagrams using SVG + CSS + a little JS.
